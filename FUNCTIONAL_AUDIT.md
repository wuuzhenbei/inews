# NewsAgg 功能真实性审查报告

> 审查日期: 2026-05-29
> 审查目标: 识别虚假操作、功能断裂、静默失败、实时性缺陷
> 结论: **系统核心链路可用，但存在大量"看起来有功能实际不能用"的问题**

---

## 审查结论速览

| 类别 | 发现数量 | 严重程度 |
|------|---------|---------|
| 虚假/不可用功能 | 4 个 | 严重 |
| 功能断裂（配置了但不生效） | 3 个 | 严重 |
| 静默失败（出错不报错） | 5 个 | 中等 |
| 实时性缺陷 | 3 个 | 中等 |
| 缺失功能 | 3 个 | 中等 |
| Bug | 3 个 | 中等 |

## 一、虚假/不可用功能

### 1.1 X/Twitter 采集器 — 几乎确定完全不能工作

**文件**: `collectors/x_collector.py`

**原因**:
- 使用 Bearer Token + 未公开的 GraphQL 端点，自 2023 年起 Twitter 已封堵此方式
- Guest Token 机制已失效
- GraphQL endpoint hash (`xmU6X_CKVnQ5lSrCbAmJsg`) 已过期，Twitter 会定期更换
- 部分账号 screen_name 有误（如 `sataborasu` 不是 Satya Nadella、`demaborul` 不是 Demis Hassabis）

**表现**: 调度器每 2 分钟调用一次 `fetch_x_tweets()`，每次都静默返回 0，用户完全不知道 55 个 X 账号一个都抓不到。

**修复方案**: 改用 RSSHub 的 Twitter 路由（`/twitter/user/{id}`），或使用 Nitter 实例的 RSS，或直接放弃 X 采集。

### 1.2 兴趣权重设置 — 前端有 UI，后端完全不生效

**文件**: `config.py:39-48`, `processors/scorer.py:128-232`

**原因**: 用户在前端设置页面可以调整 8 个方向（政治/经济/科技/军事/社会/文化/突发/财经）的权重滑块，保存到 `/api/config/weights`。但评分器 `scorer.py` 的 `_rule_based_score()` **完全没有读取这些权重**，评分逻辑是独立的关键词匹配和固定权重计算。

**表现**: 用户调整权重后感觉"没什么变化"，因为确实没有变化。

### 1.3 屏蔽关键词 — API 存在但无执行逻辑

**文件**: `app.py:37`, `app.js` (前端)

**原因**: 用户可以在设置页面添加屏蔽关键词，保存到后端。但:
- 没有任何代码读取 `blocked_keywords` 配置来过滤新闻
- `is_blocked` 字段始终为 0，没有被任何逻辑设置为 1
- 前端的 `filteredNews` 计算属性会在客户端过滤，但如果刷新页面，屏蔽词从后端加载后并未应用到查询中

**表现**: 前端即时过滤有效（同一页面内），但刷新后失效；后端查询完全不过滤。

### 1.4 市场简报按钮 — HTTP 方法不匹配，点击必报错

**文件**: `app.py` vs `static/js/app.js`

**原因**: 前端发送 POST 请求，后端只注册了 GET 方法。点击后返回 405 Method Not Allowed。

**表现**: 点击"市场简报"按钮后无反应或报错。

## 二、功能断裂（配置了但不生效）

### 2.1 AI 总结不会自动触发

**链路**: 采集 → 入库 → 评分（自动，每 120 秒） → **???** → 总结

**问题**: 评分是自动的，但 AI 总结需要用户手动点击"一键总结"按钮或调用 `/api/summary/batch`。如果不操作，新闻永远没有 AI 总结。

**影响**: 用户看到的新闻列表中，AI 总结栏大部分是空的。

**修复**: 在调度器中添加自动总结任务，或在评分完成后自动触发总结。

### 2.2 `DEFAULT_INTEREST_WEIGHTS` 配置项 — 死代码

**文件**: `config.py:39-48`

`config.py` 中定义了 `DEFAULT_INTEREST_WEIGHTS`，但整个项目中只有 `app.py` 的 `/api/config/weights` GET 端点读取它作为默认值返回。评分器不使用它，采集器不使用它，前端展示也不使用它。

### 2.3 `SCORE_THRESHOLD_TOP` 和 `DEDUP_SIMILARITY` — 死配置

**文件**: `config.py:31-32`

- `SCORE_THRESHOLD_TOP = 85` — 没有被任何代码引用
- `DEDUP_SIMILARITY = 0.85` — 没有被任何代码引用，去重完全是基于 `link` 精确匹配

## 三、静默失败（出错不报错）

### 3.1 所有采集器失败时静默返回 0

| 采集器 | 失败行为 | 用户感知 |
|--------|---------|---------|
| RSS | 单源失败记 warning，继续下一个 | 不知道哪些源失效 |
| X/Twitter | 全部失败返回 0，记一条 error | 完全无感知 |
| RSSHub | 无实例时返回 0，记一条 warning | 完全无感知 |
| 热搜 | 反爬失败返回 0，记一条 error | 完全无感知 |

**核心问题**: 用户从 Web 界面看到"没有新闻"，无法区分:
1. 确实没有新新闻
2. 所有采集器都失效了
3. 网络断了
4. API 密钥过期了

**修复方案**: 添加采集状态面板，显示每个采集器的最后运行时间、成功/失败状态、获取数量。

### 3.2 AI 评分 API 失败后静默降级到规则引擎

**文件**: `processors/scorer.py:238-241`

降级本身是好设计，但问题是用户不知道评分是来自 AI 还是规则引擎。规则引擎的评分质量明显低于 AI，但界面上看不出区别。

**修复**: 在前端显示评分来源（AI/规则引擎）。

### 3.3 RSSHub 实例不可用时静默跳过

**文件**: `collectors/rsshub_collector.py:87-88`

如果本地没有部署 RSSHub 且公共实例都不可用，`get_working_rsshub()` 返回 None，`fetch_rsshub_feeds()` 直接返回 0。用户不知道 RSSHub 相关的源全部失效。

### 3.4 `insert_news()` 竞态条件 — 未捕获的 IntegrityError

**文件**: `database/db.py:39-51`

两个线程同时插入相同 link 时，第二个 INSERT 会抛出 `sqlite3.IntegrityError`，但没有被捕获，会导致采集线程崩溃。

**修复**: 使用 `INSERT OR IGNORE` 或捕获 `IntegrityError`。

### 3.5 `get_statistics()` 时区不一致

**文件**: `database/db.py:113`

使用 `date('now')`（UTC）统计"今日新闻"，而采集器存储的是北京时间（UTC+8）。凌晨 0:00-8:00 之间统计会少算当天新闻。

对比 `get_news_list()` 使用 `date('now', 'localtime')`，两者不一致。

## 四、实时性缺陷

### 4.1 热搜采集间隔过长（15 分钟）

**文件**: `config.py:28`, `collectors/scheduler.py`

热搜数据的生命周期通常只有 30 分钟到 2 小时。15 分钟的采集间隔意味着:
- 热门话题可能在采集之前就已过时
- 排名变化无法及时捕捉
- 用户看到的"热搜"可能是 15 分钟前的状态

**建议**: 热搜采集间隔应缩短到 3-5 分钟。

### 4.2 首次启动阻塞 Web 服务

**文件**: `app.py:388-394`

`_first_collect()` 同步执行，会依次调用 4 个采集器。如果 RSSHub 连接超时（默认无超时设置），整个应用启动会卡住数分钟。

**表现**: 用户启动应用后，浏览器打开页面显示空白或连接超时，等很久才有数据。

### 4.3 RSS 源时效性参差不齐

部分 RSS 源更新频率很低:
- 国内媒体 RSS（新华网、人民网）可能几小时才更新一次
- 科技博客（GitHub Blog、AWS Blog）可能一天才更新一次
- 而调度器每 10 分钟就抓一次，造成大量无效请求

## 五、缺失功能

### 5.1 收藏功能 — 完全不存在

整个项目中搜索不到任何 bookmark/收藏相关代码:
- 数据库没有 `is_bookmarked` 列
- 没有收藏相关 API
- 前端没有收藏按钮

### 5.2 采集健康监控 — 完全不存在

没有机制让用户知道:
- 哪些采集源正在工作
- 哪些源已经失效
- 最后一次成功采集是什么时候
- 当前数据库中有多少新闻来自哪个源

### 5.3 全文搜索 — 只有 LIKE 匹配

**文件**: `database/db.py:135-141`

使用 `LIKE '%keyword%'` 全表扫描，没有 SQLite FTS5 全文索引。数据量大时性能差，且搜索结果按评分排序而非相关度。

未捕获 `IntegrityError`，可能导致采集线程崩溃。详见 3.4。

UTC vs 北京时间，凌晨统计不准。详见 3.5。

## 八、什么不是问题

以下是审查中确认**真正可用**的功能:

- RSS 采集（约 60-70% 的源有效）
- AI 评分（有完善的规则引擎降级）
- AI 总结（真实调用，有缓存）
- SSE 流式聊天（单条新闻 + 全局 RAG）
- 筛选/排序/搜索
- 统计面板（除时区问题外）
- 设置页面（四个标签页均有后端支持）
- 数据清理（每 24 小时执行一次）
- 前端无硬编码假数据

**总结**: 系统不是"假的"，但有 4 个功能是"摆设"（X 采集、兴趣权重、屏蔽关键词、市场简报），5 个地方静默失败不报错，3 个实时性缺陷。核心的 RSS 采集 → 入库 → AI 评分 → 展示 链路是通的。
